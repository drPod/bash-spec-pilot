# Literature Index

Curated literature for the **man-page → LLM → Rust impl + Bash test suite** experiment (`cp`, `mv`, `find`, `sudo`). §1–§6 are external prior art; §7 holds project-scope foundational documents (Astrogator and Aaron's prelim proposal), previously kept at the repo root, consolidated here 2026-05-30.

All papers below are indexed in delphi (arXiv ID or local PDF); query with `mcp__synsci-delphi__search_papers`.

## 1. NL → formal-spec / autoformalization

### `endres_2024_postconditions.pdf`
Endres, M., Fakhoury, S., Chakraborty, S., & Lahiri, S. K. (2024). *Can Large Language Models Transform Natural Language Intent into Formal Method Postconditions?* PACMSE / FSE 2024. [arXiv 2310.01831]

Closest analog to half of the student's task: takes English docstrings → formal postconditions (assertions). Methodologically the right reference for "extracting behavior from NL." Reports postconditions catch 64 real Defects4J bugs but also gives concrete failure modes (over-/under-specification, vacuous postconditions) that the student should expect to mirror in man-page extraction.

### `wu_2022_autoformalization.pdf`
Wu, Y., Jiang, A. Q., Li, W., Rabe, M. N., Staats, C., Jamnik, M., & Szegedy, C. (2022). *Autoformalization with Large Language Models.* NeurIPS 2022. [arXiv 2205.12615]

Origin paper for "LLMs can plausibly translate informal NL → formal artifacts." 25.3% perfect translation rate on Isabelle math sets a sober prior — most translations need repair. Tells the student to budget for a similar pass-then-repair pipeline rather than expecting one-shot success.

### `cosler_2023_nl2spec.pdf`
Cosler, M., Hahn, C., Mendoza, D., Schmitt, F., & Trippel, C. (2023). *nl2spec: Interactively Translating Unstructured Natural Language to Temporal Logics with Large Language Models.* CAV 2023. [arXiv 2303.04864]

Methodological template for handling NL ambiguity in formalization: decompose, surface candidate sub-translations to a human, iterate. Useful if the student wants an interactive ablation rather than one-shot generation, and as the canonical citation for "ambiguity is the dominant failure mode in NL → formal."

## 2. LLM code-generation correctness & failure taxonomies

### `liu_2023_evalplus.pdf`
Liu, J., Xia, C. S., Wang, Y., & Zhang, L. (2023). *Is Your Code Generated by ChatGPT Really Correct? Rigorous Evaluation of Large Language Models for Code Generation.* NeurIPS 2023. [arXiv 2305.01210]

Methodological warning: HumanEval test suites under-specify so badly that pass@k drops 19–29% under EvalPlus's 80x test inflation. Directly relevant — if the student lets the LLM generate the test suite for its own implementation (or co-evaluates impl against the LLM's tests), they will overestimate correctness. Argues for differential testing against `/bin/cp` rather than self-tests alone.

### `dou_2024_whats_wrong.pdf`
Dou, S. et al. (2024). *What's Wrong with Your Code Generated by Large Language Models? An Extensive Study.* [arXiv 2407.06153]

Direct prior art on the "failure taxonomy" deliverable. Three categories, ten sub-categories of LLM code bugs across 9 LLMs and 3 benchmarks. Student should adopt or extend this taxonomy rather than invent one — comparability matters.

### `tambon_2025_bugs_in_llm_code.pdf`
Tambon, F. et al. (2025). *Bugs in Large Language Models Generated Code: An Empirical Study.* Empirical Software Engineering. [arXiv 2403.08937]

Concrete 10-pattern bug taxonomy from 333 manually-classified LLM bugs (Misinterpretations, Silly Mistake, Prompt-biased code, Missing Corner Case, Hallucinated Object, etc.). Validated via survey with 34 practitioners. Directly the categories the student should report against; "Misinterpretations" and "Missing Corner Case" are the failure modes most likely to dominate when extracting from man pages.

### `zhang_2025_llm_hallucinations.pdf`
Zhang, Z., Wang, Y., Wang, C., Chen, J., & Zheng, Z. (2025). *LLM Hallucinations in Practical Code Generation: Phenomena, Mechanism, and Mitigation.* PACMSE 2025. [arXiv 2409.20550]

Hallucination-focused taxonomy distinct from generic-bug taxonomies: fabricated APIs, fabricated arguments, plausible-but-wrong references. Highest-priority risk when extracting from man pages — the LLM will invent flags that don't exist or attribute behavior from one utility to another.

## 3. NL → Bash specifically

### `lin_2018_nl2bash.pdf`
Lin, X. V., Wang, C., Zettlemoyer, L., & Ernst, M. D. (2018). *NL2Bash: A Corpus and Semantic Parser for Natural Language Interface to the Linux Operating System.* LREC 2018. [arXiv 1802.08979]

Foundational dataset (~10k NL+bash pairs from StackOverflow). Pre-LLM baseline (seq2seq). Useful as a corpus the student could mine for "what bash invocations users actually intend" but tells you almost nothing about modern LLM performance — cite as origin only.

### `westenfelder_2025_nl2sh.pdf`
Westenfelder, F., Hemberg, E., Tulla, M., Moskal, S., O'Reilly, U.-M., & Chiricescu, S. (2025). *LLM-Supported Natural Language to Bash Translation.* NAACL 2025. [arXiv 2502.06858]

Modern NL2Bash with **direct relevance to the test suite half**: introduces a functional-equivalence heuristic (execute + LLM-judge of outputs) reaching 95% confidence — and explicitly notes that GPT-4-as-judge alone fails because LLMs cannot emulate command execution. The student should adopt this execute-and-diff methodology for evaluating LLM-generated Bash tests against `/bin/cp`.

## 4. Prompt engineering reference

### `schulhoff_2024_prompt_report.pdf`
Schulhoff, S., Ilie, M., Balepur, N., Kahadze, K., Liu, A., Si, C., Li, Y., Gupta, A., et al. (2024). *The Prompt Report: A Systematic Survey of Prompt Engineering Techniques.* [arXiv 2406.06608, v6 Feb 2025]

PRISMA-style systematic survey of 1,565 prompting papers, distilled into a taxonomy of 58 text-prompting techniques + a controlled vocabulary of 33 terms. Use as the canonical reference when designing the impl/test prompt templates: lets us name what we are doing (e.g. "zero-shot CoT", "self-consistency", "structured output via schema") instead of inventing ad-hoc terminology, and lets a reviewer check whether we picked a sensible technique for the man-page-extraction setting. We are operating in the *single-shot, temperature=0, no human-in-the-loop* regime — sections on zero-shot, decomposition, and self-criticism are most relevant; multi-turn / agentic sections can be skipped.

## 5. Specifications + formal semantics for shell utilities

### `greenberg_2019_smoosh.pdf`
Greenberg, M., & Blatt, A. J. (2019). *Executable Formal Semantics for the POSIX Shell (Smoosh).* PACMPL/POPL 2020. [arXiv 1907.05308]

State-of-the-art hand-written formal semantics for the shell language itself (not the utilities). Establishes feasibility for shell formalization and the conformance-testing methodology (run against POSIX/Modernish suites). Tells the student: hand-written semantics for the shell exist, but **utility-level semantics (`cp`, `find`, etc.) remain almost entirely informal** — which is precisely the gap their experiment probes.

### `caruca_2025_spec_mining.pdf`
Lamprou, E., Jung, S.-H., Keoliya, M., Lazarek, L., Kallas, K., Greenberg, M., & Vasilakis, N. (2025). *Caruca: Effective and Efficient Specification Mining for Opaque Software Components.* [arXiv 2510.14279]

**Most important paper for the student to read first.** Caruca uses LLMs to translate Unix-command documentation into structured invocation syntax, then concretely executes commands with syscall/filesystem interposition to extract pre/post-conditions and properties (parallelizability, etc.). Evaluated on 60 GNU coreutils/POSIX/third-party commands; correct specs for 59/60. This is essentially the "spec-half" of the student's experiment, **already done and published**, including some of the same target utilities. Critical to read for: (1) overlap with the student's contribution, (2) what failure modes Caruca already catalogued, (3) where the student's distinct angle (executable Rust impl + behavioral test suite, not just specs) actually lies.

## 6. Lab-internal sibling work

### `slmfix_2026_emnlp.pdf`
Fu, J., Gupta, A., Councilman, A., Grove, D., Wang, Y.-X., & Adve, V. (EMNLP 2026 submission, anonymized for review; the earlier arXiv 2511.19422v1 was previously mirrored at the repo root and removed 2026-05-30 in favor of this superseding version).

**Same lab (UIUC + IBM), shared author with the student's supervisor (Aaron Councilman on both bylines).** SLMFix fine-tunes a 500M-parameter small language model via reinforcement learning (GRPO) to repair statically-detected errors (syntax, types) in LLM-generated DSL code. Two-stage reward: static validator pass + AST similarity to ground truth. Evaluated on Ansible, Bash, SQL, and Lean. Reports 40% pass-rate gain on low-resource DSLs (Ansible, Lean), 50%+ static-error elimination on high-resource DSLs (Bash, SQL). Outperforms supervised fine-tuning even on 7B-parameter base models for LRPLs.

**Direct relevance to wave 3.** Round-2 iteration in this experiment produced compile-fail regressions in 3 of 4 utilities — Rust type mismatch (`find`), macro use-before-definition (`sudo`), platform-specific syscall constants (`mv`'s `RENAME_EXCHANGE` on macOS host). SLMFix is purpose-built for exactly this failure class. Candidate use: insert SLMFix's SLM as a static-validator-first fix pass between rounds N and N+1, so the round-N+1 LLM only sees semantic feedback on code that already compiles.

**Adjacent methodological points.** (a) SLMFix uses AST-similarity (75%+ predictive of execution match per Liang et al. 2025 / Song et al. 2024) as a semantic score when no test oracle exists. This experiment has a real-binary oracle so doesn't need it as primary, but could use it as a cheap intermediate signal for ranking N≥3 resamples without spinning Docker each time. (b) SLMFix critiques test-suite-as-judge ("necessarily far from complete"). The differential-testing-against-real-binary setup here side-steps that critique because GNU coreutils is the oracle, not an LLM-generated test suite — worth naming explicitly in any cross-positioning write-up.

## 7. Project-scope foundational

These two are the parent context for the whole subproject, not external prior art. Moved into this folder 2026-05-30 from the repo root for hygiene.

### `councilman_2025_astrogator.pdf`
Councilman, A., et al. (2025). *Towards Formal Verification of LLM-Generated Code from Natural Language Prompts.* [arXiv 2507.13290]

The system this experiment extends. Verifier checks LLM-generated Ansible against a formal query the user confirms; accepts 82.9% of correct programs, rejects 92.4% of incorrect ones (Sec. 6.3). The taxonomy.md verifier-result decomposition (Sec. 2) mirrors Sec. 6.3 + 7 of this paper, swapping the verifier for differential testing. Section 7.2 explicitly names Bash as a future target — this experiment is upstream of that extension.

### `councilman_2025_prelim_proposal.pdf`
Councilman, A. (2025, preliminary). *Formal verification of LLM-generated programs — preliminary proposal.* UIUC PhD prelim.

Chapters 4 and 5 are the directly relevant ones for this experiment: Chapter 4 sketches the planned Bash extension to Astrogator (the verifier infrastructure that consumes the specs this experiment is about producing), and Chapter 5 names the **bootstrapping problem** — verifier needs formal utility specs, hand-writing each takes person-weeks, LLM-generated specs may themselves be wrong. The "use unreliable LLMs to produce reliable specs" framing in the project README comes directly from this chapter.
